VOTEWISE – AI-POWERED ELECTION ASSISTANT (GOOGLE STACK)
MASTER DEVELOPER PROMPT
=============================================================

You are an expert full-stack developer and AI engineer. Your task
is to build "VoteWise" — a production-ready, AI-powered Election
Assistant web application built exclusively on Google services.
This application helps citizens — especially first-time voters —
understand elections, check eligibility, and navigate the voting
process through an intelligent chatbot and interactive UI.

Your solution will be evaluated on:
  • Code Quality       – structure, readability, maintainability
  • Security           – safe and responsible implementation
  • Efficiency         – optimal use of Google resources
  • Testing            – functional validation at every layer
  • Accessibility      – WCAG-compliant, inclusive design
  • Google Integration – meaningful, non-trivial use of Google services

=============================================================
SECTION 1 — OBJECTIVE
=============================================================

Build a complete, deployable web application that:
  1. Guides users through the election process step-by-step
  2. Answers natural-language election queries via a Gemini-powered
     chatbot (eligibility, documents, EVMs, booth procedures, etc.)
  3. Provides first-time voter onboarding with visual walkthroughs
  4. Displays election timelines, phases, and deadlines
  5. Checks voter eligibility based on user-provided inputs
  6. Delivers a searchable FAQ system backed by Firestore
  7. Sends push notifications for key election dates (optional tier)

FEATURE 1 │ SMART CHATBOT (Core Feature)
─────────────────────────────────────────

Cloud Function: `chatHandler` (HTTPS Callable)

  System prompt to inject before every user message:
  ┌─────────────────────────────────────────────────────────┐
  │ You are VoteWise, an expert, friendly, and neutral      │
  │ election assistant for Indian voters. You only answer   │
  │ questions related to elections, voting procedures,      │
  │ voter registration, eligible documents, EVMs, NOTA,     │
  │ election phases, and civic rights. For off-topic        │
  │ questions, politely redirect the user to election       │
  │ topics. Keep answers concise (under 150 words),         │
  │ factual, and beginner-friendly. Use bullet points       │
  │ where helpful. Never express political bias.            │
  └─────────────────────────────────────────────────────────┘

  Implementation steps:
    a. Accept { userMessage, history[] } from client
    b. Validate input — reject empty or >500 char messages
    c. Reconstruct conversation history array for Gemini
    d. Call Gemini 1.5 Pro via @google/generative-ai SDK
    e. Return { reply: string } to client
    f. Save both turns to Firestore `chatHistory` subcollection
    g. Log token usage to Firebase Analytics custom event

    • Require Firebase Auth token on every callable function
    • Rate-limit: max 30 requests/user/hour (use Firestore counter)
    • Sanitize all inputs server-side before sending to Gemini
    • Never expose the Gemini API key on the client

  Sample Cloud Function skeleton:
  ─────────────────────────────────
  exports.chatHandler = onCall(async (request) => {
    if (!request.auth) throw new HttpsError('unauthenticated', ...);
    const { userMessage, history } = request.data;
    // Input validation
    // Build Gemini message array with system prompt
    // Call Gemini API
    // Save to Firestore
    // Return reply
  });

FEATURE 4 │ ELIGIBILITY CHECKER
─────────────────────────────────────────

  Cloud Function: `checkEligibility` (HTTPS Callable)

  Input form fields:
    • Age (number input, min 1, max 120)
    • Citizenship (dropdown: Indian Citizen / Other)
    • Residence status (Indian resident / NRI)

  Logic (server-side in Cloud Function):
    if (citizenship !== "Indian" || age < 18) → NOT ELIGIBLE
    else if (residenceStatus === "NRI")       → ELIGIBLE (special NRI rules apply)
    else                                      → ELIGIBLE

  Output card:
    ✅ Eligible   →  Show next steps (register, find booth)
    ❌ Not Eligible → Explain reason + show when they will qualify

  Save result to `users/{uid}.eligibilityResult` in Firestore.

SECTION 8 — SECURITY REQUIREMENTS
=============================================================

  • Gemini API key: stored only in Firebase Functions environment
    (`firebase functions:config:set gemini.key="..."`)
    Never hardcoded, never shipped to client bundle
  • Firestore rules: deny all by default; explicitly allow per route
  • Input validation: all user inputs validated both client and server
  • Rate limiting: Cloud Function call counts tracked in Firestore
  • Auth required: all Cloud Functions check `request.auth`
  • HTTPS only: Firebase Hosting enforces HTTPS automatically
  • No PII logged: strip user-identifiable data from Analytics events
  • Content moderation: Gemini safety settings set to BLOCK_MEDIUM_AND_ABOVE

=============================================================
SECTION 9 — CODE QUALITY STANDARDS
=============================================================

  • React: functional components + hooks only (no class components)
  • Folder structure:
      /src
        /components   → reusable UI components
        /pages        → route-level page components
        /hooks        → custom hooks (useAuth, useFirestore, useChat)
        /services     → firebase.js, gemini.js, fcm.js
        /utils        → validators, formatters, constants
        /styles       → global CSS + design tokens
      /functions
        /src          → Cloud Function handlers (one file per feature)
  • JSDoc comments on all exported functions
  • PropTypes or TypeScript interfaces on all components
  • No hardcoded strings — use constants file for all labels
  • Async/await with try-catch on every Firebase and API call
  • Meaningful error messages surfaced to the user (not raw errors)
